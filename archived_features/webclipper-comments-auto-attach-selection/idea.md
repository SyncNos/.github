# webclipper-comments-auto-attach-selection · idea

## 背景 / 触发

- 当前 inpage 评论引用的主入口依赖“选中文本 + 双击 inpage 按钮”打开侧栏后带入 `quoteText`。
- 这个入口操作负担偏高，且与用户在右侧评论侧栏中的心智不一致（用户已在评论上下文中，却还要回到按钮双击）。
- 本次需求经过方案取舍后确定 pivot：不新增显式按钮，改为“点击根评论输入框自动附加当前选区”。

## 核心需求（原始需求精炼）

1. 在 **inpage + App/Popup** 两端，点击根评论输入框（`Write a comment…`）时，自动把当前选区附加到引用区。
2. 自动附加仅作用于根评论输入框；回复输入框（`Reply…`）禁止触发自动附加。
3. 每次点击根评论输入框都按“当前选区”覆盖引用区（而非仅首填）。
4. 点击时若当前无有效选区，则清空引用区。
5. 选区来源语义保持现有行为：
   - inpage 沿用页面选区语义；
   - App/Popup 沿用当前消息区域选区语义。
6. 兼容现有打开入口：双击 inpage 按钮、popup 按钮、context menu 打开评论侧栏流程不回归。

## 默认值与兼容策略

- 新旧用户统一默认开启该行为（不新增设置开关，不引入迁移步骤）。
- 保持现有 `quoteText + locator` 数据结构与持久化路径，不调整 schema 与同步协议。
- 后续若需要可扩展“可配置开关”，但本 feature 不引入该扩展。

## 非目标（明确不做什么）

- 不改回复输入框交互。
- 不重做评论面板布局与视觉风格。
- 不改评论同步（Notion/Obsidian）协议与渲染逻辑。
- 不在本 feature 内引入新的设置项或实验开关。

## 验收标准（可检查）

- 在 inpage 与 App/Popup 场景，点击根评论输入框：
  - 有选区时，引用区显示为当前选区并覆盖旧值；
  - 无选区时，引用区被清空。
- 点击任意回复输入框不会改变引用区。
- 保存根评论后，引用清空行为与当前实现保持一致（不新增脏状态）。
- 双击 inpage 按钮仍可打开侧栏并可继续评论；popup/context menu 入口功能不退化。
- `compile + test` 通过，新增/更新测试覆盖 inpage 与 App/Popup 的自动附加主路径与边界行为。
