# AI Chat 会话目录式导航面板

## 背景 / 触发

- AI Chat 对话通常很长（可达几万字），标题与内容不总是对应；用户回溯定位很痛苦。
- 需要一个“目录式导航”，让用户快速浏览对话结构并跳转到对应位置。

## 核心需求（原始需求精炼）

1. **目录条目**：每条 `role=user` 的消息 = 1 个目录条目，按对话顺序编号（从 1 开始）。
2. **条目文本**：使用消息的纯文本（优先 `contentText`，否则从 `contentMarkdown` 提取纯文本）取前 30 字 + `…`，单行显示（超出截断）。
3. **触发与位置**：
   - 在会话详情页内容区右上角**常驻**一个“目录把手”（两道横线），把手**贴右侧**、**无边框/无阴影**、不呈现按钮视觉。
   - hover 把手时，从右侧以 **overlay** 形式滑出目录面板（Notion 目录风格），并且打开时目录面板**覆盖把手**。
   - 仅把手区域触发（不保留“右侧边缘热区”）。
4. **跳转**：点击目录条目后，详情页平滑滚动到对应的用户消息（尽量居中）。
5. **自动高亮**：滚动对话时，目录自动高亮“最接近视口中线”的用户消息条目。
6. **作用范围**：
   - 仅在 WebClipper 的本地 UI（`app.html` + popup 的会话详情页）生效。
   - 仅对 `conversation.sourceType === 'chat'` 的会话显示（严格以此为真源）。
   - 纯前端实现，零配置，不依赖任何 LLM 接入。

## 默认值与兼容策略

- 默认开启：对所有 `sourceType='chat'` 会话详情页都显示目录把手与面板。
- 不做历史兜底：若未来出现 `sourceType` 缺失但实际是 chat 的老数据，本 feature 不负责推断；作为独立 bug 再修。

## 非目标（明确不做什么）

- 不注入原始 AI chat 网页（不做 content script inpage 目录）。
- 不做“按 assistant 消息分段 / 按标题自动分段”等 LLM/语义分段能力。
- 不做跨会话聚合目录、搜索、过滤等高级能力。

## 验收标准（可检查）

- 在 `conversation.sourceType !== 'chat'`（例如 article）时：详情页不出现目录把手。
- 在 `conversation.sourceType === 'chat'` 时：
  - 详情页内容区右上角常驻目录把手；hover 把手后出现目录面板；打开时面板覆盖把手；移出面板后收回。
  - 目录条目数量 = `detail.messages` 中 `role === 'user'` 的消息数；编号连续且顺序正确。
  - 点击任意条目：详情滚动平滑跳转并将对应 user bubble 尽量置于视口中线附近。
  - 用户滚动详情：目录高亮跟随，始终高亮“最接近视口中线”的 user 条目。
