# webclipper-comments-selection-recovery · idea

## 背景 / 触发

- 当前评论侧栏引用读取依赖原生 `globalThis.getSelection()`。
- 在得到app网页版等站点，页面使用自定义选中 UI，导致原生 Selection 为空，插件无法识别用户实际选中内容。
- 已确认范围：仅评论侧栏引用链路；白名单站点先行；不做全插件改造。

## 核心需求（原始需求精炼）

1. 在白名单站点上，当原生 Selection 为空时，评论侧栏引用链路应尝试“坐标反推”恢复选中文本。
2. 反推逻辑仅作用于评论根输入框引用附加流程；回复输入框行为不变。
3. 反推成功时应尽量同时产出 `quoteText + locator`；若 locator 失败但文本可得，允许 `locator = null`。
4. 反推失败必须静默降级，不影响面板打开、评论保存与既有流程。
5. 非白名单站点保持现有行为，避免引入全局副作用。
6. 兼容 Chrome/Edge 主链路；Firefox 不支持时保持现状降级。

## 默认值与兼容策略

- 默认启用“白名单站点坐标反推兜底”，首批 host：`www.dedao.cn`、`dedao.cn`。
- 非白名单不触发反推，行为与当前版本一致。
- 后续可扩展更多 host 或 page-world 适配器，但首版不引入设置项，不改存储 schema。

## 非目标（明确不做什么）

- 不改造 comments 以外的选区消费模块。
- 不使用“解劫持页面事件/CSS”方案。
- 不引入 OCR / 截图权限链路。
- 不在首版实现站点 page-world 深度注入适配。

## 验收标准（可检查）

- 在首批白名单站点中，原生 Selection 为空但文本在 DOM 的场景，根评论输入框可附加有效引用文本。
- 反推方向（左->右、右->左）都可得到稳定文本，跨段落选区可用。
- 若 locator 构建失败，保存根评论时仍能保存 `quoteText` 且 `locator` 为空。
- 回复输入框不会触发附加或清空引用。
- `compile + test + build` 通过，新增用例覆盖 host gate、坐标反推、降级路径。
