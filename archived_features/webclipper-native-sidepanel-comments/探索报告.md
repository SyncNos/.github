# 探索报告：在 WebClipper 右侧侧边栏内嵌原生 AI 网页（iframe）

日期：2026-03-26  
仓库：SyncNos / `webclipper/`  
范围：`.github/features/webclipper-native-sidepanel-comments`

## 摘要（结论先行）

我们尝试在 **现有 inpage 右侧 Comments 侧边栏**中新增 `AI chats` 模式，用 `iframe` 直接加载 ChatGPT / Notion AI 等“原生 AI 网页”，并尝试用 MV3 `declarativeNetRequest`（DNR）剥离响应头来绕过 `X-Frame-Options`/`CSP frame-ancestors`。

最终结论是：**“在任意网页中 embed 原生 AI 站点”无法做成一个稳定、可发布的能力**。即使偶尔能加载，也会反复遇到：

- 宿主页面 CSP（`frame-src`/`child-src`）直接拦截（代表：GitHub）
- 被嵌站点的防嵌策略（`X-Frame-Options` + `frame-ancestors`）
- 登录/会话在 iframe 场景不稳定（第三方 Cookie、Storage Access API 不可控）
- Passkey/WebAuthn 在 iframe 内经常不可用（或站点强制 top-level）
- 风控/反滥用：iframe + 非典型嵌入环境更容易触发封禁/屏蔽
- 商店审核风险：DNR 剥安全响应头属于“安全绕过”，高概率不通过

因此决定：**放弃 “inpage iframe embed 原生 AI 网页” 作为 P1 方向**。

我们随后在 `crh2` 分支验证了 **P2：AI 伴随 tab（顶层页面）**（为每个内容 tab 创建/复用一个 AI companion tab，并 best-effort move + tab group 绑定）。结论是：**技术上可行，且“同一内容 tab 复用同一 AI tab”生效，但产品价值不符合预期**——用户反馈“这与原本已存在的 Chat with AI 按钮相比没有本质区别”，因此 **P2 同样停止推进**（本 feature 方向整体失败）。

## 目标与动机

### 最初目标

- 不做 Chrome “原生 side panel”（因为其跨 tab 常驻，不满足 per-tab 映射）
- 复用现有的 inpage 右侧 comments sidebar
- 用 header 下拉在 `Comments` / `AI chats` 之间切换
- `AI chats` 默认加载用户在 Settings 里配置的第一个 AI 平台

### 为什么想要 iframe

- 直接复用原生站点 UI（完整功能、无需自研聊天 UI）
- 期待保留登录态（Cookie/session）
- 希望成本最低地实现“像分屏一样的体验”

## 实验路径（做过什么）

### 1) 共享侧边栏（Comments / AI chats）

实现方式：在原有 comments panel 内增加 mode 与下拉切换，并在 `AI chats` 下渲染 iframe。

价值：验证了“共享一个侧边栏容器 + header 下拉切换”的 UI/交互是可行的。

### 2) 方案 B：DNR 硬剥响应头（尝试绕过防嵌）

核心思路：用 MV3 `declarativeNetRequest` 的 `modifyHeaders` 在浏览器网络层移除：

- `X-Frame-Options`
- `Content-Security-Policy`（含 report-only 与 legacy 头）

目标：让 ChatGPT / Notion 等在 iframe 中“强行可加载”。

我们用 `curl -I https://www.notion.so/` 验证过 Notion 确实包含：

- `x-frame-options: SAMEORIGIN`
- `content-security-policy: ... frame-ancestors 'self' ...`

因此理论上“如果 DNR 生效”，至少能过掉其中一层阻拦。

## 关键障碍（为什么最终放弃）

### A. 宿主页面 CSP（frame-src/child-src）是硬拦截，无法靠 DNR 解决

现象（代表案例：GitHub 页面）：

浏览器报错类似：

> Framing 'https://www.notion.so/' violates the following Content Security Policy directive: "frame-src ...". The request has been blocked.

解释：

- 这是 **当前网页（宿主页面）** 的 CSP 限制了它页面里允许嵌入的 frame 来源
- 属于“请求前拦截”，与 Notion 的响应头无关
- 我们可以对 Notion 响应头做任何事，但 **宿主页面根本不让嵌入** → iframe 永远加载不了

结论：只要我们把入口放在“任意网页注入的 inpage sidebar”，就必然碰到这一类站点，无法做成通用能力。

### B. 被嵌站点的防嵌策略（XFO/CSP）本身就是对抗目标

即使在宿主页面 CSP 允许的站点里：

- Notion/ChatGPT 仍可能通过 `frame-ancestors`/`X-Frame-Options`/其它 runtime 检测阻止嵌入
- DNR 剥头是“对抗式绕过”，且可能在不同浏览器版本/不同响应链路下不稳定

### C. 登录态与 Cookie：iframe 下的会话不可靠（且不可控）

即使 iframe 加载成功：

- Cookie 在跨域 iframe 往往被视为第三方 Cookie（受 SameSite、浏览器策略影响）
- Safari/Firefox/Brave 等默认更严格
- `Storage Access API` 需要 iframe 内站点配合，我们无法控制 Notion/ChatGPT 的实现

结果：登录/跳转/验证可能随机失败，无法承诺“像原生一样工作”。

### D. Passkey / WebAuthn：iframe 内经常不可用

实际观察：即使 iframe 能进站点页面，**浏览器的 passkey 流程仍可能无法正常弹出/完成**。

原因：

- 需要 Permissions Policy 委托（iframe `allow`）才“可能”可用
- 但很多站点仍强制 top-level 上下文（防钓鱼/风控）

结论：即使局部修补，也无法保证 passkey 这类登录体验。

### E. 风控与封禁：嵌入环境更容易触发站点防护

现象：出现“该内容被屏蔽了/频繁被 ban”的反馈。

合理推断：

- iframe 嵌入 + 异常导航 + 非典型 referer/permissions 环境，会被站点视为可疑
- 这属于对方业务规则/风控策略，扩展难以稳定对抗

### F. Chrome Web Store / Edge Add-ons 审核风险

用 DNR 去剥安全响应头，本质是“削弱网页自我防护机制”。这类能力：

- 很可能被认定为危险/绕过安全策略
- 即使通过一次，也存在后续被下架/整改的风险

结论：不适合作为正式发布功能。

## 决策

**放弃**：在 inpage sidebar 中 `iframe` embed 原生 AI 网页（ChatGPT/Notion 等）作为产品能力。  
**放弃**：P2（per-tab AI companion tab）作为替代路线（原因见下文 P2 复盘）。  
**保留可复用成果**：共享侧边栏的 UI 结构（header 下拉模式切换）可以继续作为入口壳，但“AI 内容载体”需要换成非 iframe、且不是“仅打开一个 tab”的形态。

## 推荐后续方案（替代路线）

### 方案 P2（已验证但否决）：AI 伴随 tab（顶层页面）

要点：

- 不用 iframe，直接打开/复用一个 AI tab（顶层 context）
- 建立 `contentTabId -> aiTabId` 映射
- 将两个 tab 通过 tab group 绑定，并尽量相邻排序
- 触发时在两个 tab 间跳转（不是同屏，但稳定）

优点：

- 登录态、passkey、站点能力都按“正常浏览器使用方式”工作
- 不受宿主页面 CSP `frame-src` 限制
- 无需 DNR 安全绕过，审核风险显著降低

缺点（本次否决原因）：

- 体验本质仍是“打开一个 AI tab 并在两个 tab 间跳转”，无法形成“右侧侧边栏同屏”
- 与原先已存在的 `Chat with AI` 按钮相比，用户感知到的差异不足（即使实现了 per-tab 复用）

P2 实测证据（crh2，2026-03-26）：

在 Chrome 扩展 Service Worker 控制台中可观察到 per-tab 复用生效（同一 source tab 复用同一 `aiTabId`）：

```
[SyncNos] OPEN_COMPANION_AI_TAB result: { aiTabId: 1141700439, created: true, url: 'https://www.notion.so/ai' }
[SyncNos] OPEN_COMPANION_AI_TAB result: { aiTabId: 1141700439, created: false, url: 'https://www.notion.so/ai' }
```

### 其它可选

- 扩展 App 页面（`chrome-extension://` 顶层）内嵌：理论上能避开“宿主页面 CSP”，但仍会被“被嵌站点 XFO/CSP + 登录 + passkey”等问题卡住
- 自研聊天 UI + 官方 API：体验与成本不同，是另一条产品路线

## 对代码/实现的影响（建议）

由于该方向确认放弃，建议后续做一次清理：

- 移除/禁用 `AI chats` iframe 相关 UI
- 移除 DNR “剥安全响应头”实验代码与权限（避免审核/安全风险）
- 将侧边栏 header 下拉保留为“入口容器”，但不要继续强化 “iframe embed” 或 “companion tab” 方向；应回到更符合产品目标的入口与载体设计
