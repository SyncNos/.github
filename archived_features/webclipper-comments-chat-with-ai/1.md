# 评论级 Chat with AI：从收集工具到思考入口

## 背景

SyncNos 当前的 Chat with AI 是**文章级别**的：用户在文章详情页点击按钮，把整篇文章作为上下文跳转到 AI 平台。这个功能已经上线。

但用户真正关心的往往不是整篇文章，而是**文章中的某一段 + 自己对这一段的想法**。这恰好就是评论侧边栏里每条 root comment 已经在做的事——引用原文片段 + 写下评论。

## 目标

在 inpage 评论侧边栏的每条 **root comment** 上增加一个 **Chat with AI** 按钮，让用户可以**带着精准上下文**（引用原文 + 评论内容）直接跳转到 AI 平台展开深度对话。

## 非目标

- 不做同屏嵌入 AI（已验证不可行，见「在有侧边栏，加入嵌入AI网页的功能」）
- 不做自研聊天 UI / API 调用
- 不改变现有的文章级 Chat with AI 行为
- reply（子回复）暂不加此按钮，只做 root comment

## 用户故事

1. 用户在网页上阅读文章
2. 划线选中一段文字，通过评论侧边栏写下自己的想法（root comment）
3. 在这条评论旁边看到 Chat with AI 按钮
4. 点击后，SyncNos 自动拼装上下文（引用原文 + 评论正文），按用户在设置中配置的 AI 平台跳转
5. AI 平台收到的不是 5000 字全文，而是一段精准的引用 + 用户的思考起点

## 上下文 Payload 设计

拼装规则（复用 `chatwith-settings.ts` 的模板逻辑）：

```
我在阅读一篇文章时，对其中一段内容有一些想法，想和你深入讨论。

---
📎 引用原文：
{quoteText}

💬 我的想法：
{commentText}

---
文章标题：{articleTitle}
文章来源：{canonicalUrl}
```

- `quoteText` 和 `commentText` 从 comment 数据模型中直接取
- `articleTitle` 和 `canonicalUrl` 从当前 article 会话中取
- 字符截断逻辑复用现有 `chatwith-settings.ts` 的限制

## 技术方案

### 核心认知：共享渲染器

`threaded-comments-panel.ts` 是 `ArticleCommentsSection.tsx`（app/popup 详情页）和 `inpage-comments-panel-shadow.ts`（页面内侧边栏）的**共享渲染器**。按钮只需加在 `threaded-comments-panel.ts` 一处，两个入口自动都有。

### 已有可复用的基础设施

| 模块 | 文件 | 复用点 |
| --- | --- | --- |
| Chat with AI 配置与模板 | `chatwith-settings.ts` | 平台列表、prompt 模板变量、最大字符数（`chat_with_max_chars_v1`）、复制载荷逻辑 |
| Chat with AI 执行链路 | `chatwith-detail-header-actions.ts` | 「复制 payload → 打开平台首页 tab」的跳转执行逻辑（需从中抽取共享函数） |
| 评论数据模型 | `comments/data/storage-idb.ts` | `quoteText`、`commentText`、`canonicalUrl`、`conversationId` |
| 评论渲染器 | `threaded-comments-panel.ts` | root comment（`parentId === null`）的渲染与按钮交互 |

### ⚠️ 不复用的（容易搞混）

- **`detail-header-action-types.ts`**：定义的是详情页头部的 `open / chat-with / tools` 槽位系统，跟评论级按钮无关，不需要碰
- **`message-contracts.ts`**：不新增消息类型，按钮动作在 UI 层完成（Phase 4 的 tab group 会新增消息类型）

### 关键架构问题：inpage 上下文的能力差异

在 app/popup 中，可以直接 import `chatwith-settings.ts` 读取配置。但在 **inpage shadow DOM（content script）** 中：

- 读取 `chrome.storage.local` 的 chatwith 配置（`chat_with_ai_platforms_v1` 等）→ content script 可以直接读，**没问题**
- 执行剪贴板写入 → `navigator.clipboard.writeText()` 在 shadow DOM 中可能受限，需验证；若不行，走 background messaging
- 打开新 tab → content script 无法直接 `chrome.tabs.create()`，**必须走 background messaging**

**结论**：inpage 场景下，按钮点击后应发送消息到 background，由 background 执行「复制 + 开 tab」。app/popup 场景可以直接执行。建议跳转函数内部判断运行环境，自动选择直接执行或走 messaging。

### 边界情况

- **root comment 没有 quoteText**（用户直接写评论没有划线引用）：payload 中省略引用原文段，只保留评论内容 + 文章元信息
- **article 处于 orphan 状态**（尚无 `conversationId`）：`articleTitle` 和 `canonicalUrl` 从 `comment-sidebar-session.ts` 的 shared session 或当前页面 DOM 中取，不依赖 conversation 必须已存在

### 不需要改动的

- 存储层（不改 comment 数据结构）
- 备份/恢复链路
- 文章级 Chat with AI 逻辑（detail header 的现有行为不变）

## 实施阶段

### Phase 1：复制上下文到剪贴板

用户点击评论旁的 Chat with AI 按钮后，评论内容被复制到剪贴板，用户可以手动粘贴到任何 AI 平台。

- 用户体验：点击 → toast 提示“已复制” → 用户自行切换到 AI 平台粘贴
- 已有能力：文章级 Chat with AI 已实现同样的复制流程

### Phase 2：自动跳转到 AI 平台

复制完成后自动打开用户配置的 AI 平台新标签页，减少一步手动操作。

- 用户体验：点击 → 复制 payload → 自动打开 AI 平台新标签页 → 用户粘贴开聊
- 已有能力：文章级 Chat with AI 已有同样的跳转流程

### Phase 3：评论级精准上下文

发送给 AI 的不再是整篇文章，而是**引用原文 + 用户想法 + 文章元信息**的精准组合。

- 用户体验：AI 收到的是“这段话 + 我的想法”，而不是 5000 字全文
- 边界：没有划线引用的评论，只发送评论内容 + 文章元信息

### Phase 4：同一篇文章复用同一个 AI 对话

用户对同一篇文章的多条评论点击 Chat with AI，都跳转到**同一个 AI 聊天窗口**，形成连续的讨论上下文。

- 用户体验：第一次点击开新 tab，后续点击回到同一个 tab，评论上下文自然累积
- 新增权限：需要 `chrome.tabGroups` 权限

### Phase 5：自动填入 AI 输入框并发送（暂不实现）

跳转到 AI 平台后，自动把上下文填入输入框并发送，用户无需手动粘贴。

- 实现复杂度高：涉及各 AI 平台输入框适配
- **先不实现，等 Phase 1–4 稳定后再评估**

## 验收标准

- [ ]  inpage 评论侧边栏中，每条 root comment 显示 Chat with AI 按钮
- [ ]  点击按钮后，正确拼装 `quoteText + commentText + articleTitle + canonicalUrl` 为 prompt
- [ ]  按用户设置跳转到对应 AI 平台（复用现有跳转逻辑）
- [ ]  reply 评论不显示此按钮
- [ ]  payload 字符数超限时正确截断
- [ ]  app 内 article detail 的 `ArticleCommentsSection` 中同样支持此按钮

## 产品意义

这个 feature 把评论侧边栏从**记录层**升级为**思考入口**：

> 读文章 → 划线评论 → 带着这条想法跟 AI 深聊
> 

用户存下来的东西不再是死的笔记，而是可以随时激活的思考起点。这是 SyncNos 从「收集工具」走向「Personal Context Layer」的第一步。
